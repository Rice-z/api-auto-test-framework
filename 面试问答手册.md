# 接口自动化测试框架 — 面试问答手册

---

## 一、项目相关（必问）

### Q1：介绍一下你的自动化测试项目

> 我用 Python + pytest + requests 从零搭了一个接口自动化测试框架。整体是分层架构——config 管配置、common 管工具封装、data 管 YAML 测试数据、testcases 管用例执行。核心亮点有三个：第一，通过 conftest.py 的 session 级别 fixture 实现全局自动登录，整个测试过程只登录一次，Token 注入后所有用例自动携带认证；第二，用 YAML 数据驱动 + pytest parametrize 实现数据和代码分离，加新用例只加数据不改代码；第三，内置了一个用 Python http.server 写的 Mock API 服务器，不依赖外部网络也能跑。跑完后 pytest-html 自动生成 HTML 报告。目前覆盖了登录 4 种场景加 2 个认证接口，6 条用例全部通过。

---

### Q2：为什么用分层架构？

> 四个字：关注点分离。配置变了只改 settings.yaml，被测接口变了只改 YAML 数据，底层能力在 common 里统一维护，加新业务只需在 testcases 加新文件。每层职责清晰，互不干扰，方便维护和扩展。

---

### Q3：conftest.py 的作用是什么？

> conftest.py 是 pytest 的特殊文件，放在项目根目录，里面定义的 fixture 会被同级及子目录下所有测试用例**自动发现**，不需要显式 import。
>
> 我的 conftest 做了三件事：
> 1. 加载 settings.yaml 全局配置
> 2. 创建全局 HTTP 客户端（requests.Session）
> 3. **自动登录并注入 Token**——这是最核心的
>
> 三个 fixture 形成依赖链：`base_url → req_client → login_token`。

---

### Q4：全局自动登录是怎么实现的？

> 在 conftest.py 里定义了一个 `login_token` fixture，scope 设成 session：
> ```python
> @pytest.fixture(scope="session")
> def login_token(req_client):
>     # 1. 发登录请求
>     resp = req_client.post("/api/v1/login", json={"username": "admin", "password": "123456"})
>     # 2. 提取 token
>     token = resp.json().get("token", "")
>     # 3. 注入公共请求头——关键步骤
>     req_client.update_headers({"Authorization": f"Bearer {token}"})
>     return token
> ```
> session 级别意味着整个测试会话只执行一次。之后 testcases 里所有用例引用的 req_client 都已经携带了这个 Token。

---

### Q5：pytest fixture 的 scope 有哪几种？

| scope | 执行频率 | 适用场景 |
|-------|---------|---------|
| function（默认） | 每个测试函数执行一次 | 需要隔离的用例 |
| class | 每个测试类执行一次 | 类内共享数据 |
| module | 每个 .py 文件执行一次 | 模块内共享 |
| **session** | 整个测试过程执行一次 | 全局登录、数据库连接 |

> 我的 login_token 用 session，因为整个测试只登录一次就够了，不需要每个用例都登录。

---

### Q6：什么是数据驱动测试？你怎么实现的？

> 数据驱动测试就是把测试数据和测试逻辑分离开。我的做法：
> 1. 测试数据写在 YAML 文件里，每条用例包含 name（用例名）、data（请求参数）、expected（预期结果）
> 2. 测试代码里用 `@pytest.mark.parametrize` 遍历这些数据
> 3. 1 个测试方法 + N 条 YAML 数据 = N 条独立用例，在报告里各自显示
>
> 好处：加用例只加数据，不改代码。非技术人员也能在 YAML 里加测试场景。

---

### Q7：`@pytest.mark.parametrize` 怎么用的？

```python
@pytest.mark.parametrize("case", login_data, ids=[c["name"] for c in login_data])
def test_login(self, case, req_client):
    resp = req_client.post("/api/v1/login", json=case["data"])
    AssertUtil.assert_status_code(resp, case["expected"]["code"])
    AssertUtil.assert_contains(resp, "msg", case["expected"]["msg"])
```
> - 第一个参数 `"case"` 是变量名
> - 第二个参数 `login_data` 是数据列表
> - `ids` 给每条用例起名字，出现在测试报告里
> - pytest 会自动把 4 条数据变成 4 条独立用例

---

### Q8：为什么用 `requests.Session()` 而不是直接 `requests.get()`？

> Session 对象会**自动保持 Cookie 和请求头**。普通 `requests.get()` 每次都是独立请求，Cookie 不会自动延续。
>
> Session 的好处：
> 1. Cookie 自动管理——登录后的 Session Cookie 后续请求自动携带
> 2. 公共请求头统一管理——通过 `update_headers()` 一次性注入，所有请求生效
> 3. 连接池复用——底层 TCP 连接可以复用，性能更好
> 4. 全局配置一次生效——base_url、timeout 等集中管理

---

### Q9：Mock 服务器是干什么的？

> Mock 服务器模拟了真实的被测系统。用 Python 内置的 `http.server` 模块写的，零额外依赖。
>
> 它做的事情：
> - POST /api/v1/login → 校验用户名密码，正确返回 token，错误返回 401/400
> - GET /api/v1/users → 检查 Authorization 请求头，有 token 才返回数据
> - GET /api/v1/user/profile → 同上
>
> 好处：
> 1. 不依赖外部网络，离线也能跑
> 2. 可以精确控制接口行为（想让它返回什么就返回什么）
> 3. 面试时能直接演示

---

### Q10：你怎么生成测试报告？

> 用 pytest-html 插件，在 run.py 里配置：
> ```python
> pytest_args = [
>     "--html=reports/report.html",
>     "--self-contained-html",     # CSS 内嵌，一个文件就能看
>     "--junitxml=reports/report.xml",  # CI 工具可解析
> ]
> ```
> HTML 报告给人类看，XML 给 Jenkins 等 CI 工具解析。

---

### Q11：如果让你加一个"获取用户列表"的测试，怎么做？

> 三步，不用改任何已有代码：
> 1. 在 data/ 下建 `user_list.yaml`，写测试数据（路径、参数、预期结果）
> 2. 在 testcases/ 下建 `test_user_list.py`，写测试类和方法
> 3. 方法参数里声明 `req_client` 和 `login_token`，pytest 自动注入已登录的客户端
>
> 因为 conftest 的全局登录已经处理好了认证，新用例直接复用就行。

---

### Q12：你这个框架有什么不足/可以改进的地方？

> 1. 目前没有并发执行——可以加 pytest-xdist 实现多线程并行
> 2. 没有失败重试机制——可以加 pytest-rerunfailures
> 3. 目前测试数据是写死的，可以改成从 Excel/数据库读取
> 4. 报告没有发送通知——可以集成钉钉/企业微信机器人
> 5. 可以加 CI/CD Pipeline，代码提交自动触发测试

## 二、Python 基础（必问）

### Q13：`if __name__ == "__main__"` 是干什么的？

> 当用 `python run.py` 直接运行时，`__name__` 的值是 `"__main__"`，会执行 main() 函数。
> 但当别人 `import run` 时，`__name__` 是 `"run"`，不会执行。防止被 import 时意外触发。

---

### Q14：装饰器是什么？pytest 里哪里用了？

> 装饰器本质是一个函数，它接收另一个函数作为参数，在不修改原函数的前提下增加额外功能。
>
> 我的项目里：
> ```python
> @pytest.fixture(scope="session")   # 这是装饰器，把普通函数变成 pytest fixture
> @pytest.mark.parametrize(...)      # 这也是装饰器，给测试函数加参数化能力
> ```

---

### Q15：单例模式是什么？你的项目里哪用了？

> 单例模式保证一个类全局只有一个实例。我的 log_util.py 里：
> ```python
> def __new__(cls):
>     if cls._instance is None:
>         cls._instance = super().__new__(cls)
>     return cls._instance
> ```
> 重写 `__new__` 方法，第一次创建实例后保存到 `_instance`，以后再调用都返回同一个对象。避免日志文件被重复打开、日志重复输出。

---

### Q16：`*args` 和 `**kwargs` 有什么区别？

> - `*args`：接收任意数量的**位置参数**，打包成**元组**
> - `**kwargs`：接收任意数量的**关键字参数**，打包成**字典**
>
> 我的 requests_util.py 里 `_request(self, method, path, **kwargs)` 用了 **kwargs，把额外参数透传给 `requests.Session.request()`。

## 三、HTTP 协议（必问）

### Q17：GET 和 POST 有什么区别？

| | GET | POST |
|------|-----|------|
| 参数位置 | URL 后面（Query String） | 请求体（Body） |
| 安全性 | 参数暴露在 URL，不适合传密码 | 参数在 Body，相对安全 |
| 长度限制 | URL 长度有限制（约 2KB） | 请求体无限制 |
| 缓存 | 可以被浏览器缓存 | 不会被缓存 |
| 幂等性 | 幂等（多次请求结果相同） | 非幂等（每次可能创建新资源） |
| 用途 | 查数据 | 提交数据、登录 |

---

### Q18：HTTP 常见状态码有哪些？

| 状态码 | 含义 | 举例 |
|--------|------|------|
| 200 | OK，请求成功 | 登录成功返回 token |
| 201 | Created，创建成功 | 新增用户成功 |
| 301/302 | 重定向 | 跳转到登录页 |
| 400 | Bad Request，客户端请求错误 | 用户名或密码为空 |
| 401 | Unauthorized，未认证 | 没登录或 token 过期 |
| 403 | Forbidden，无权限 | 没有访问权限 |
| 404 | Not Found，资源不存在 | 接口路径写错了 |
| 500 | Internal Server Error，服务器错误 | 后端代码崩了 |
| 502 | Bad Gateway，网关错误 | 上游服务挂了 |
| 503 | Service Unavailable，服务不可用 | 服务器在维护 |

---

### Q19：Cookie 和 Token 有什么区别？

| | Cookie | Token |
|------|--------|-------|
| 存储位置 | 浏览器自动管理 | 客户端手动存储（localStorage/内存） |
| 跨域支持 | 受同源策略限制 | 天然支持跨域 |
| 适用场景 | Web 浏览器 | Web + App + 小程序 |
| 状态 | 服务端需维护 Session | 服务端无状态 |

> 我的框架用 Token（Bearer Token），放在请求头 Authorization 里。

---

### Q20：HTTPS 和 HTTP 有什么区别？

> HTTPS = HTTP + SSL/TLS 加密层。
> - HTTP 是明文传输，中间人可以窃听、篡改
> - HTTPS 加密传输，保证数据机密性、完整性
> - HTTPS 需要 CA 证书验证服务器身份
> - HTTPS 默认端口 443，HTTP 默认 80

## 四、测试理论（高频）

### Q21：黑盒测试和白盒测试有什么区别？

> - **黑盒测试**：不看代码，只关注输入和输出是否匹配需求。比如功能测试、接口测试。
> - **白盒测试**：看代码内部逻辑，关注代码覆盖率。比如单元测试、代码走查。
> - **灰盒测试**：介于两者之间，知道部分内部逻辑来做测试设计。

---

### Q22：等价类和边界值是什么？举个例子。

> **等价类**：把输入划分为若干等价区间，每个区间取一个代表值测试。
> **边界值**：重点测试等价类边界上的值，因为边界最容易出 bug。
>
> 举例——测试密码长度限制 6~20 位：
> - 等价类：有效类(6~20位)、无效类(<6位)、无效类(>20位)
> - 边界值：5位、6位、20位、21位、0位
>
> 我的 YAML 数据里，空用户名、空密码就属于边界值测试。

---

### Q23：什么是回归测试？什么时候做？

> 回归测试就是代码修改后，重新跑一遍已有用例，确认**新改动没有破坏旧功能**。
>
> 什么时候做：每次提测/上线前、Bug 修复后、代码重构后。自动化测试最大的价值就在回归测试——每次改完代码跑一遍，省去大量手工重复劳动。

---

### Q24：冒烟测试和回归测试有什么区别？

> - **冒烟测试**：只测核心流程能不能走通，时间短（几分钟），用于判断版本值不值得继续测
> - **回归测试**：全量验证已有功能，时间长，确保没有引入新 bug
>
> 冒烟是"能不能用"，回归是"有没有坏"。

---

### Q25：什么场景适合自动化测试？什么不适合？

> **适合的：**
> - 接口测试（稳定、高频、数据驱动）
> - 回归测试（反复执行的用例）
> - 数据校验、批量操作
> - 冒烟测试（核心流程快速验证）
>
> **不适合的：**
> - UI 频繁变动的功能
> - 只跑一次的探索性测试
> - 需要人主观判断的（如图片美观度、用户体验）
> - 没有明确预期的场景

---

### Q26：一个登录页面你怎么设计测试用例？

> 功能测试：
> - 正确账号密码 → 登录成功，跳转首页
> - 错误密码 → 提示"用户名或密码错误"
> - 空用户名 → 提示"用户名不能为空"
> - 空密码 → 提示"密码不能为空"
> - 记住密码功能是否生效
> - 密码是否显示为 ***
>
> 安全测试：
> - 密码错误 5 次是否锁定账号
> - SQL 注入：输入 `admin'--` 是否能绕过
> - XSS：输入 `<script>alert(1)</script>` 是否弹窗
>
> 性能测试：
> - 大量用户同时登录响应时间是否正常
>
> 兼容测试：
> - Chrome/Firefox/Edge 不同浏览器
> - iOS/Android 移动端

## 五、数据库（常见）

### Q29：SQL 连表查询 JOIN 有哪几种？

| JOIN 类型 | 结果 |
|-----------|------|
| INNER JOIN | 两表都匹配的行 |
| LEFT JOIN | 左表全部行 + 右表匹配的行（没匹配的填 NULL） |
| RIGHT JOIN | 右表全部行 + 左表匹配的行 |
| FULL JOIN | 两表全部行（MySQL 不直接支持） |

```sql
-- 查每个用户的订单数
SELECT u.name, COUNT(o.id)
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.name;
```

---

### Q30：GROUP BY 和 HAVING 有什么区别？

> GROUP BY 用于分组，HAVING 用于过滤**分组后的结果**。
> WHERE 在分组前过滤，HAVING 在分组后过滤。
>
> ```sql
> -- 查订单数 > 3 的用户
> SELECT user_id, COUNT(*) as cnt
> FROM orders
> GROUP BY user_id
> HAVING cnt > 3;
> ```

### Q33：git merge 和 git rebase 有什么区别？

> - merge：保留分支历史，产生一个合并提交。适合公共分支。
> - rebase：把当前分支的提交"移到"目标分支后面，历史更线性干净。适合个人分支。
>
> 经验法则：公开分支用 merge，私有分支用 rebase。

## 七、场景题（考察测试思维）

### Q35：给你一个水杯怎么测？

> - 功能：能不能装水、倒水、盖上盖子、保温
> - UI：外观是否完整、logo 是否清晰
> - 容量：标注 500ml 实测是否准确
> - 边界：0ml 装不装得了、超过 500ml 会不会溢出
> - 异常：倒热水会不会烫手、摔地上会不会碎
> - 兼容：不同液体（水/咖啡/碳酸饮料）
> - 性能：保温 6 小时后温度下降多少

---

### Q36：微信发朋友圈怎么测？

> - 功能：文字/图片/视频/混合发送、@好友、定位、可见范围
> - 异常：断网发送、超大图片、超长文字、发送中退出
> - 兼容：iOS/Android、不同系统版本、不同屏幕尺寸
> - 性能：9 张大图加载速度、列表滑动流畅度

---

### Q37：一个接口响应时间突然变慢了，你怎么排查？

> 1. 看服务器资源：CPU、内存、磁盘 IO 有没有异常
> 2. 看数据库：慢查询日志、索引是否失效、连接池是否满了
> 3. 看网络：ping、traceroute 看延迟
> 4. 看代码：最近有没有上线、有没有新增复杂查询
> 5. 看并发量：是不是突然流量激增
> 6. 用 Postman/Charles 抓包看是哪个环节慢
